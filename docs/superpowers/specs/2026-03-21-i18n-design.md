# Internationalization (i18n) Design Spec

## Goal

Add multi-locale support to the static Astro portfolio site with three markets: `en-us` (default), `en-gb`, and `de-de`. Each market has fully independent content. Locale detection redirects first-time visitors once and remembers their choice.

## Locales

| Locale  | Language | URL prefix | Default |
|---------|----------|------------|---------|
| `en-us` | English (US) | `/` (none) | Yes |
| `en-gb` | English (GB) | `/en-gb/` | No |
| `de-de` | German | `/de-de/` | No |

All three markets are treated as fully independent — different text, pricing (including currency), testimonials, case studies. Each locale's engagement pricing uses its own currency symbol and amounts directly in the UI strings (e.g., `$150/hour` for en-us, `£120/hour` for en-gb, `€130/Stunde` for de-de).

## URL Structure

```
/                        → en-us homepage
/cases/[id]              → en-us case detail
/en-gb/                  → en-gb homepage
/en-gb/cases/[id]        → en-gb case detail
/de-de/                  → de-de homepage
/de-de/cases/[id]        → de-de case detail
```

## Astro i18n Configuration

```js
// astro.config.mjs
i18n: {
  locales: ['en-us', 'en-gb', 'de-de'],
  defaultLocale: 'en-us',
  routing: {
    prefixDefaultLocale: false,
  },
}
```

Uses Astro's built-in i18n routing. `en-us` lives at `/` without prefix. Non-default locales get prefixed paths. `prefixDefaultLocale: false` is the default but stated explicitly to avoid ambiguity.

## Shared Layout Component

Create `src/layouts/Layout.astro` to eliminate duplication across the six page files. It contains:

- `<html lang={htmlLang}>` with locale-appropriate lang attribute
- `<head>` with meta tags, Google Fonts links, IBM Plex Mono font-face declarations
- Theme detection inline script (light/dark mode)
- Locale redirect inline script (conditionally, only on default locale pages)
- `hreflang` link tags for all locale variants
- Localized `<title>` and `<meta name="description">`
- Global CSS (resets, theme variables, reveal animations)
- `<slot />` for page content

Props: `locale`, `title`, `description`, `hreflangAlternates` (array of `{ locale, href }`), `includeRedirectScript` (boolean, default false).

## Translation Architecture

### UI Strings

One TypeScript file per locale in `src/i18n/`:

```
src/i18n/
├── en-us.ts       # All UI strings for en-us
├── en-gb.ts       # All UI strings for en-gb
├── de-de.ts       # All UI strings for de-de
├── types.ts       # Shared type definition for string objects
└── utils.ts       # Helpers
```

Each locale file exports a typed object containing all translatable strings: navigation labels, hero content (headline, pitch, role metadata, CTA text, image alt text), section headings, CTAs, testimonials, engagement pricing (with currency), situations list, footer text, contact form labels, and page `<title>` / meta description.

`types.ts` defines the `Strings` interface so all locale files stay in sync. A missing key is a type error.

`utils.ts` exports:
- `getStrings(locale: Locale)` — returns the `Strings` object for a locale
- `type Locale = 'en-us' | 'en-gb' | 'de-de'`
- `locales` — array of all supported locales
- `defaultLocale` — `'en-us'`

Note: Use `Astro.currentLocale` (built into Astro's i18n) to get the current locale in `.astro` files at build time. The `getStrings()` helper is for mapping that locale to the corresponding strings object. No custom URL parser is needed server-side.

### Long-form Content (Case Studies)

Locale subdirectories in the content collection:

```
src/content/cases/
├── en-us/
│   ├── saas-platform.md
│   ├── analytics-engine.md
│   ├── ddp-app.md
│   └── mobile-dashboard.md
├── en-gb/
│   └── ...  (market-specific cases)
├── de-de/
│   └── ...  (market-specific cases)
```

Each locale's cases are fully independent — different titles, descriptions, even different case studies per market. Case slugs do NOT need to match across locales.

A content collection config (`src/content.config.ts`) must be created or updated to define the schema for cases (title, subtitle, image, etc.).

### Case Detail getStaticPaths

Since content entry IDs include the locale prefix (e.g., `en-us/saas-platform`), each locale's case page filters and strips the prefix:

```ts
// src/pages/cases/[id].astro
export async function getStaticPaths() {
  const cases = await getCollection('cases', ({ id }) => id.startsWith('en-us/'));
  return cases.map((entry) => ({
    params: { id: entry.id.replace('en-us/', '') },
    props: { entry },
  }));
}

// src/pages/de-de/cases/[id].astro — same pattern with 'de-de/' prefix
```

## Component Changes

All components currently have hardcoded English text. They must accept a `strings` prop (or the relevant section of it). Components can read `Astro.currentLocale` directly for locale-aware URL generation (via `getRelativeLocaleUrl`), so a separate `locale` prop is not needed.

### Pattern

```astro
---
import type { Strings } from '../i18n/types';
import { getRelativeLocaleUrl } from 'astro:i18n';

type Props = {
  nav: Strings['nav'];
};
const { nav } = Astro.props;
const locale = Astro.currentLocale!;
---
<a href={getRelativeLocaleUrl(locale, '/')}>{nav.logo}</a>
```

Components that need updating:
- `navigation.astro` — logo text, CTA label, language picker, aria-labels
- `hero.astro` — headline, pitch, role metadata, CTA text, image alt text
- `logo-section.astro` — group labels, company names (may differ per market)
- `situations-section.astro` — heading, situation items
- `proof-section.astro` — section labels, case data, testimonials, engagement pricing (with currency)
- `contact-section.astro` — heading, form labels, placeholder text
- `footer.astro` — copyright text, link labels

## Page Structure

### Homepage

```
src/pages/
├── index.astro              # en-us
├── en-gb/
│   └── index.astro          # en-gb
├── de-de/
│   └── index.astro          # de-de
```

Each locale page is a thin wrapper: imports `Layout`, calls `getStrings()` with the locale, and passes string sections to each component. All layout and head logic lives in `Layout.astro`.

### Case Detail Pages

```
src/pages/
├── cases/
│   └── [id].astro           # en-us cases
├── en-gb/cases/
│   └── [id].astro           # en-gb cases
├── de-de/cases/
│   └── [id].astro           # de-de cases
```

Each filters `getCollection('cases')` by its locale prefix and strips the prefix from the ID for the URL param.

## Language Picker

Located in the navigation, next to the theme toggle and CTA.

### Appearance

- Shows current locale's flag (emoji) + language name
- On click, expands a dropdown showing all three options with flags
- Each option links to the target locale's homepage

### Flag Mapping

| Locale | Flag | Label |
|--------|------|-------|
| en-us | US flag | English (US) |
| en-gb | GB flag | English (GB) |
| de-de | DE flag | Deutsch |

### Behavior

- On the homepage: links point to `/{target-locale}/`
- On case detail pages: links point to the target locale's homepage (since case slugs are independent per market, there is no guaranteed equivalent page)
- Stores the chosen locale in `localStorage` key `locale`

## Client-side Redirect

An inline `<script is:inline>` in the `<head>`, included only on default locale pages (`/` and `/cases/*`) via the `Layout` component's `includeRedirectScript` prop.

### Logic

```
1. Read localStorage.getItem('locale')
2. If stored locale exists → do nothing (user already chose or was already redirected)
3. If no stored locale:
   a. Read navigator.language (e.g., "de", "de-DE", "en-GB")
   b. Map to supported locale:
      - starts with "de" → "de-de"
      - matches "en-gb" (case-insensitive) → "en-gb"
      - anything else → "en-us" (already here, store and stop)
   c. Store mapped locale in localStorage
   d. If mapped locale ≠ en-us → redirect to /{mapped-locale}/ (always redirect to homepage, not deep link, to avoid 404s from non-matching case slugs)
```

This runs only on the default locale's pages. Visitors arriving at `/de-de/` or `/en-gb/` directly are never redirected.

## SEO

### hreflang Tags

Every page includes `<link rel="alternate">` tags for its locale variants. Handled by `Layout.astro`:

Homepage:
```html
<link rel="alternate" hreflang="en-US" href="https://sebastian-heitmann.dev/" />
<link rel="alternate" hreflang="en-GB" href="https://sebastian-heitmann.dev/en-gb/" />
<link rel="alternate" hreflang="de-DE" href="https://sebastian-heitmann.dev/de-de/" />
<link rel="alternate" hreflang="x-default" href="https://sebastian-heitmann.dev/" />
```

Case detail pages: only include hreflang for the current page (no cross-locale alternates, since case slugs are independent per market).

### HTML Lang Attribute

Set by `Layout.astro` based on the `locale` prop:
- `<html lang="en-US">` on en-us pages
- `<html lang="en-GB">` on en-gb pages
- `<html lang="de-DE">` on de-de pages

### Title and Meta Description

Each locale's `Strings` object includes `meta.title` and `meta.description`. These are passed to `Layout.astro` for the `<title>` and `<meta name="description">` tags.

## What's NOT Changing

- Theme toggle (light/dark/system) — works independently of locale
- Visual design / CSS — identical across all locales
- Component file structure — same components, just parameterized with strings
- Build output — still fully static
