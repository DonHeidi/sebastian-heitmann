# Internationalization (i18n) Design Spec

## Goal

Add multi-locale support to the static Astro portfolio site with three markets: `en-us` (default), `en-gb`, and `de-de`. Each market has fully independent content. Locale detection redirects first-time visitors once and remembers their choice.

## Locales

| Locale  | Language | URL prefix | Currency | Default |
|---------|----------|------------|----------|---------|
| `en-us` | English (US) | `/` (none) | USD | Yes |
| `en-gb` | English (GB) | `/en-gb/` | GBP | No |
| `de-de` | German | `/de-de/` | EUR | No |

All three markets are treated as fully independent — different text, pricing, testimonials, case studies.

## URL Structure

```
/                        → en-us homepage
/cases/[id]              → en-us case detail
/en-gb/                  → en-gb homepage
/en-gb/cases/[id]        → en-gb case detail
/de-de/                  → de-de homepage
/de-de/cases/[id]        → de-de case detail
```

## Astro i18n Configuration

```js
// astro.config.mjs
i18n: {
  locales: ['en-us', 'en-gb', 'de-de'],
  defaultLocale: 'en-us',
}
```

Uses Astro's built-in i18n routing. `en-us` lives at `/` without prefix. Non-default locales get prefixed paths.

## Translation Architecture

### UI Strings

One TypeScript file per locale in `src/i18n/`:

```
src/i18n/
├── en-us.ts       # All UI strings for en-us
├── en-gb.ts       # All UI strings for en-gb
├── de-de.ts       # All UI strings for de-de
├── types.ts       # Shared type definition for string objects
└── utils.ts       # Helper: t(locale), getLocaleFromUrl()
```

Each locale file exports a typed object containing all translatable strings: navigation labels, hero content, section headings, CTAs, testimonials, engagement pricing, situations list, footer text, and contact form labels.

`types.ts` defines the shape so all locale files stay in sync. A missing key is a type error.

`utils.ts` exports:
- `t(locale: Locale)` — returns the string object for a locale
- `getLocaleFromUrl(url: URL)` — extracts locale from the current URL path
- `type Locale = 'en-us' | 'en-gb' | 'de-de'`
- `locales` — array of all supported locales
- `defaultLocale` — `'en-us'`

### Long-form Content (Case Studies)

Locale subdirectories in the content collection:

```
src/content/cases/
├── en-us/
│   ├── saas-platform.md
│   ├── analytics-engine.md
│   ├── ddp-app.md
│   └── mobile-dashboard.md
├── en-gb/
│   └── ...  (market-specific cases)
├── de-de/
│   └── ...  (market-specific cases)
```

Each locale's cases are fully independent — different titles, descriptions, even different case studies per market.

## Component Changes

All components currently have hardcoded English text. They must accept a `strings` prop (or subset of it) and a `locale` prop.

### Pattern

```astro
---
// Component receives its section's strings
type Props = {
  locale: Locale;
  nav: Strings['nav'];
};
const { locale, nav } = Astro.props;
---
<a href={getRelativeLocaleUrl(locale, '/')}>{nav.logo}</a>
```

Components that need updating:
- `navigation.astro` — logo text, CTA label, language picker
- `hero.astro` — headline, pitch, role metadata, CTA text
- `logo-section.astro` — group labels, company names (may differ per market)
- `situations-section.astro` — heading, situation items
- `proof-section.astro` — section labels, case data, testimonials, engagement pricing
- `contact-section.astro` — heading, form labels, placeholder text
- `footer.astro` — copyright text, link labels

## Page Structure

### Homepage

```
src/pages/
├── index.astro              # en-us: imports t('en-us'), passes to components
├── en-gb/
│   └── index.astro          # en-gb: imports t('en-gb'), passes to components
├── de-de/
│   └── index.astro          # de-de: imports t('de-de'), passes to components
```

Each locale page imports the same components and passes locale-specific strings. The page files are thin wrappers — all layout logic stays in components.

### Case Detail Pages

```
src/pages/
├── cases/
│   └── [...slug].astro      # en-us cases
├── en-gb/cases/
│   └── [...slug].astro      # en-gb cases
├── de-de/cases/
│   └── [...slug].astro      # de-de cases
```

Each uses `getCollection('cases')` filtered by locale prefix in the content path.

## Language Picker

Located in the navigation, next to the theme toggle and CTA.

### Appearance

- Shows current locale's flag (emoji) + language name (e.g., "English (US)")
- On click/hover, expands a dropdown showing all three options with flags
- Each option links directly to the equivalent page in that locale

### Flag Mapping

| Locale | Flag | Label |
|--------|------|-------|
| en-us | US flag | English (US) |
| en-gb | GB flag | English (GB) |
| de-de | DE flag | Deutsch |

### Behavior

- Clicking a locale navigates to the equivalent page (e.g., `/de-de/` if on `/`, or `/de-de/cases/foo` if on `/cases/foo`)
- Stores the chosen locale in `localStorage` key `locale`

## Client-side Redirect

An inline `<script is:inline>` in the `<head>` of the default locale's pages (`/` and `/cases/*` only — not on `/en-gb/` or `/de-de/` pages).

### Logic

```
1. Read localStorage.getItem('locale')
2. If stored locale exists → do nothing (user already chose)
3. If no stored locale:
   a. Read navigator.language (e.g., "de", "de-DE", "en-GB")
   b. Map to supported locale:
      - starts with "de" → "de-de"
      - "en-gb" or "en-GB" → "en-gb"
      - anything else → "en-us" (already here, store and stop)
   c. If mapped locale ≠ en-us → redirect to /{mapped-locale}/{current-path}
   d. Store mapped locale in localStorage
```

This runs only on the default locale's pages. Visitors arriving at `/de-de/` directly are never redirected.

## SEO

### hreflang Tags

Every page includes `<link rel="alternate">` tags for all locale variants:

```html
<link rel="alternate" hreflang="en-US" href="https://sebastian-heitmann.dev/" />
<link rel="alternate" hreflang="en-GB" href="https://sebastian-heitmann.dev/en-gb/" />
<link rel="alternate" hreflang="de-DE" href="https://sebastian-heitmann.dev/de-de/" />
<link rel="alternate" hreflang="x-default" href="https://sebastian-heitmann.dev/" />
```

### HTML Lang Attribute

- `<html lang="en-US">` on en-us pages
- `<html lang="en-GB">` on en-gb pages
- `<html lang="de-DE">` on de-de pages

## What's NOT Changing

- Theme toggle (light/dark/system) — works independently of locale
- Visual design / CSS — identical across all locales
- Component file structure — same components, just parameterized with strings
- Build output — still fully static
