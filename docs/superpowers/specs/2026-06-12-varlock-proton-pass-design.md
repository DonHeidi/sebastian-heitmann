# varlock + Proton Pass Secret Management — Design

**Date:** 2026-06-12
**Status:** Approved (pending spec review)

## Goal

Replace the repo's plaintext `.env` files with [varlock](https://github.com/dmno-dev/varlock) schema-driven config across all workspaces, and source every genuinely-sensitive value from a [Proton Pass](https://protonpass.github.io/pass-cli/) vault via the `pass-cli`. No secret ever lands on disk in plaintext.

## Scope

- **In scope:** All env config across `apps/website`, `apps/mail-service`, `apps/job-directory`, and `infra`. Local development and the two deploy scripts (`apply-infra.sh`, `deploy-website.sh`).
- **Out of scope:** CI/CD service accounts or personal access tokens for automation; changes to how Terraform injects the function's *runtime* environment; any website-side (client) secrets (there are none).

## Why these tools fit

- **varlock** keeps a committed `.env.schema` as the single source of truth for config shape. Non-secret values are committed defaults; secrets are resolved at process start via function resolvers. Items are explicitly marked `@sensitive` rather than relying on naming prefixes.
- **Proton Pass** has a first-class varlock plugin, [`@varlock/proton-pass-plugin`](https://varlock.dev/plugins/proton-pass/). It is registered with `@plugin(@varlock/proton-pass-plugin)` and **must also be instantiated** with the `@initProtonPass()` root decorator (verified against plugin 1.0.2: `@plugin` alone yields "No Proton Pass plugin instances found"). Called with no arguments, `@initProtonPass()` creates the default instance and reuses an already-authenticated `pass-cli` session — credentials (`username`/`password`/`totp`) are only needed for non-interactive CI login, which is out of scope. The `protonPass(pass://vault/item/field)` resolver then runs `pass-cli item view --output json` and extracts the **last path segment** as the field name, so Proton Pass field names must match the reference exactly. varlock also layers any existing `.env` file over the schema, so a legacy `.env` value shadows the resolver until deleted (eases migration).

## Secret surface (established by inspection)

| Value | Today | Classification |
|-------|-------|----------------|
| `SCW_ACCESS_KEY`, `SCW_SECRET_KEY` | `apps/mail-service/.env` (sourced by `apply-infra.sh`); also `~/.config/scw/config.yaml` (read by `deploy-website.sh`) | **Secret** — deploy/infra credential |
| `SCW_DEFAULT_ORGANIZATION_ID`, `SCW_DEFAULT_PROJECT_ID` | `apps/mail-service/.env` | Sensitive identifiers |
| `JOB_DIRECTORY_API_TOKEN` | `apps/job-directory/src/server/auth.ts` reads `process.env` | **Secret** |

| `TEM_SECRET_KEY` | **Generated by Terraform** (`scaleway_iam_api_key.mail_service.secret_key`), injected into the function | **Not human-held — stays out of Proton Pass** |
| `PUBLIC_SITE_URL`, `PUBLIC_MAIL_ENDPOINT` | `apps/website/.env` | Non-secret (public) |
| `MAIL_SENDER`, `ALLOWED_ORIGINS`, `TEM_REGION` | `.env` / `.tfvars` defaults | Non-secret config |
| `JOB_DIRECTORY_DB` | default path in `connection.ts` | Non-secret config |

## Proton Pass vault layout

Single vault, **grouped by service boundary** (one item per service, related fields together — values that rotate and are scoped together live in one item):

```
Vault: sebastian-heitmann
├── item "scaleway"      → fields: access-key, secret-key, org-id
├── item "job-directory" → field:  api-token
└── item "mail"          → field:  recipient
```

Rationale: the Scaleway access/secret keys are a pair that always rotates together and shares one access scope, so they belong in one item; each *service* remains a separate item so a future personal access token can be scoped to just one service. Trade-off accepted: field names must match the schema exactly (a renamed field silently breaks resolution), mitigated by varlock validating `@required` at load.

## Architecture

```
.env.schema (committed, per directory)
        │  @plugin(@varlock/proton-pass-plugin)
        │  protonPass(pass://sebastian-heitmann/<item>/<field>)
        ▼
   varlock run  ──resolves in-memory──▶  child process env  (bun dev / terraform / astro build)
        ▲
   pass-cli session  ◀── authenticated (personal access token) ──  Proton Pass vault
```

Nothing materializes to disk. Every dev and deploy command runs behind `varlock run --`.

## Per-directory schemas

### `apps/website/.env.schema`
No secrets — purely committed defaults. No `protonPass()` refs and no plugin, so website-only contributors never need Proton Pass.
```sh
# @type=url
PUBLIC_SITE_URL=https://www.sebastian-heitmann.dev
# @type=url @required
PUBLIC_MAIL_ENDPOINT=https://your-function-endpoint.functions.fnc.fr-par.scw.cloud
```
(`PUBLIC_MAIL_ENDPOINT` is overridden at deploy time from `terraform output`, as today.)

### `apps/mail-service/.env.schema`
Local mail-service runtime config only. The `SCW_*` deploy credentials are **removed from here** (see Cleanup).
```sh
# @plugin(@varlock/proton-pass-plugin)
# ---
# @type=email
MAIL_SENDER=contact@contact.sebastian-heitmann.dev
ALLOWED_ORIGINS=https://www.sebastian-heitmann.dev,https://sebastian-heitmann.dev
TEM_REGION=fr-par
# @sensitive @type=email
MAIL_RECIPIENT=protonPass(pass://sebastian-heitmann/mail/recipient)
```

### `apps/job-directory/.env.schema`
```sh
# @plugin(@varlock/proton-pass-plugin)
# ---
# @sensitive @required
JOB_DIRECTORY_API_TOKEN=protonPass(pass://sebastian-heitmann/job-directory/api-token)
JOB_DIRECTORY_DB=./data/jobs.db
```

### `infra/.env.schema`
```sh
# @plugin(@varlock/proton-pass-plugin)
# ---
# @sensitive @required
SCW_ACCESS_KEY=protonPass(pass://sebastian-heitmann/scaleway/access-key)
# @sensitive @required
SCW_SECRET_KEY=protonPass(pass://sebastian-heitmann/scaleway/secret-key)
# @required
SCW_DEFAULT_ORGANIZATION_ID=protonPass(pass://sebastian-heitmann/scaleway/org-id)
# @sensitive @type=email
TF_VAR_mail_recipient=protonPass(pass://sebastian-heitmann/mail/recipient)
```

`SCW_DEFAULT_ORGANIZATION_ID` is supplied so the `scw` CLI authenticates purely from env (see Deployment portability) — its `scw account project list` lookup then needs no `~/.config/scw/config.yaml`. The project ID itself stays dynamically resolved via that lookup, so no `project-id` field is required in Proton Pass.

> **Implementation note:** confirm the exact `@plugin`/`@initProtonPass` header form against the installed plugin version, and that local resolution reuses the authenticated `pass-cli` session without requiring `@initProtonPass()` credentials.

## Delivery: the `varlock run` wrapper

- **package.json scripts** gain a `varlock run --` prefix, e.g. `apps/website` `"dev": "varlock run -- astro dev"`, `"build": "varlock run -- astro build"`; equivalents for `mail-service` and `job-directory`.
- **`scripts/apply-infra.sh`**: remove the `SCW_ACCESS_KEY`/`SCW_SECRET_KEY` presence check and the "source `apps/mail-service/.env`" guidance. Run Terraform as `cd "$INFRA_DIR" && varlock run -- terraform apply "$@"`. The `AWS_ACCESS_KEY_ID="${SCW_ACCESS_KEY}@${PROJECT_ID}"` derivation stays in bash (varlock supplies the `SCW_*` halves into the child env; the `scw account project list` lookup for `PROJECT_ID` is unchanged). `mail_recipient` reaches Terraform as `TF_VAR_mail_recipient` and is removed from `terraform.tfvars`.
- **`scripts/deploy-website.sh`**: replace the `grep`/`awk` over `~/.config/scw/config.yaml` with `varlock run` supplying `SCW_ACCESS_KEY`/`SCW_SECRET_KEY`; the `AWS_*` derivation and `terraform output` steps are unchanged.

## Deployment portability

**Goal:** any machine with the toolchain and an authenticated `pass-cli` session (vault access) can deploy — no machine-local Scaleway config required.

Both scripts run their Scaleway-touching commands under `varlock run`, which injects `SCW_ACCESS_KEY`, `SCW_SECRET_KEY`, and `SCW_DEFAULT_ORGANIZATION_ID` into the environment. The `scw` CLI reads these env vars natively, so `scw account project list` resolves the project ID without `~/.config/scw/config.yaml`. From there the existing `AWS_ACCESS_KEY_ID="${SCW_ACCESS_KEY}@${PROJECT_ID}"` derivation (Terraform state backend auth) is unchanged.

**Per-machine prerequisites (not secrets):** the toolchain is pinned via mise (see below), so setup is `mise install` + `bun install` + `pass-cli login`. The only auth step is `pass-cli` login with a personal access token scoped to the `sebastian-heitmann` vault.

## Toolchain (mise)

The repo already pins `bun` in the root `mise.toml`. This design extends it to pin the rest of the deploy toolchain so any machine resolves identical binaries:

```toml
[tools]
bun = "latest"
terraform = "<pinned exact, ≥ 1.10>"   # ≥ 1.10 required for S3-native state locking (use_lockfile)
scaleway = "<pinned exact>"            # provides the `scw` binary (aqua:scaleway/scaleway-cli)
```

- **Exact versions** are pinned (not `latest`) for reproducibility; resolve the current `terraform` and `scaleway` versions at implementation and write them in. `terraform` must stay ≥ 1.10.
- **`pass-cli`** is *not* in the mise registry, so it stays a documented install (`curl -fsSL https://proton.me/download/pass-cli/install.sh | bash`), or wired via mise's `ubi:` backend if it ships GitHub releases — verify at implementation.
- **`varlock` + `@varlock/proton-pass-plugin`** are workspace dev dependencies resolved by `bun install`; they are not pinned through mise.

## Cleanup baked into this work

`apps/mail-service/.env` currently doubles as the *deploy credential* store — `apply-infra.sh` sources `SCW_*` keys from it. That conflates an infra/deploy credential with mail-service runtime config. This design moves the `SCW_*` credentials into `infra/.env.schema` (their true home) and `mail_recipient` out of `terraform.tfvars` into Proton Pass, so no PII sits in a committed-adjacent file.

## Data flow at apply time (unchanged contract)

Terraform continues to self-generate `TEM_SECRET_KEY` and inject the function's runtime environment. varlock only feeds Terraform its provider/state-backend *auth* credentials (`SCW_*` → `AWS_*`) and the single human input (`TF_VAR_mail_recipient`).

## Error handling

- varlock validates `@required` and `@type` decorators at load, failing fast with a clear message if a reference is missing or `pass-cli` is unauthenticated — instead of a downstream 401/502 at runtime.
- `pass-cli` authenticates once via a personal access token scoped to the `sebastian-heitmann` vault; the plugin reuses that session.

## Onboarding

Documented in `AGENTS.md`:
1. `mise install` (bun, terraform, scaleway CLI) + `bun install` (adds `varlock` + `@varlock/proton-pass-plugin`).
2. Install `pass-cli` (`curl -fsSL https://proton.me/download/pass-cli/install.sh | bash`) and authenticate with a personal access token scoped to the `sebastian-heitmann` vault.
3. Create the `sebastian-heitmann` vault and the three items/fields above.
4. Run any app via its existing `bun run dev` (now wrapped); deploy via the existing scripts.
